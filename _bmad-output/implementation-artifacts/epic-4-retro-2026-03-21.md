## What Went Well

### Architectural Preparation Paid Off
`usePersistenceStore` was fully implemented and tested before Epic 4 began. Stories 4.1 and 4.2 were wiring work, not invention. This kept implementation clean and predictable. Story 4.4 was entirely tests-only because the corruption-handling implementation was already correct — a sign of quality-first development.

### Test Count Discipline
Each story added exactly the tests it needed:
- 4.1: 30 tests for new wiring
- 4.2: 14 tests for streak logic
- 4.3: 4 e2e tests (zero unit changes — infrastructure-only story)
- 4.4: 6 tests for untested corruption paths
- 4.5: 5 e2e tests for gameplay flows

No over-testing or under-testing. The test count growth was purposeful.

### E2E Policy Established
The absence of e2e tests across Epics 1–4 was a planning gap. Story 4.5 backfilled coverage for all shipped features. The formal e2e test policy was added to `epics.md` to prevent recurrence: every story with user-visible features must include Playwright tests going forward.

### Story 4.4 as a "Verification Sprint"
The pattern of shipping implementation early (4.1) and deferring comprehensive tests to a focused story (4.4) worked well. The implementation was correct; the tests confirmed it. This is a repeatable pattern for architecturally-designed components.

## What Was Hard

### Issue 1 — Mock Incompleteness (Story 4.2)
**What happened:** Three new methods were added to `usePersistenceStore`. `GameView.test.ts` was mocking the old store interface. 17 tests broke and required discovery + mock updates.

**Root cause:** No type enforcement on mock objects. Mocks are plain objects — TypeScript cannot catch missing methods.

**Impact:** Debugging time spent on test failures that should have been compile errors.

**Prevention:** Typed mocks — mock objects typed as `Partial<ReturnType<typeof useXxxStore>>` or a matching interface. TypeScript then surfaces missing methods at compile time when the store interface changes.

### Issue 2 — E2E Assertion Strength (Story 4.3)
**What happened:** Initial e2e offline gameplay test asserted that 5 tile slots exist in the DOM. This passes even if the game engine never ran. A sprint change proposal (BS-2, BS-4) was needed to strengthen assertions to check `tile-state-correct/present/absent` CSS classes — proving the engine evaluated the guess.

**Root cause:** No shared standard for what "good" e2e assertions look like. The dev wrote structurally valid tests; the QA review found they were behaviorally weak.

**Prevention:** Add to story e2e task templates: "Assertions must prove behavior ran — check state classes, emitted events, or DOM values. Do NOT assert only that elements exist in the DOM."

### Issue 3 — Node 24 ESM JSON Import Syntax (Stories 4.3 and 4.5)
**What happened:** Node 24 requires `with { type: 'json' }` on JSON imports in ESM context. This was discovered at runtime in story 4.3, then rediscovered in story 4.5. The dev notes for 4.3 did not document the finding.

**Root cause:** Lessons discovered during implementation were not propagated to project-wide dev notes or architecture documentation.

**Prevention:** Project-level gotchas (Node version quirks, ESM requirements, framework-specific patterns) belong in the architecture doc, not only in individual story debug logs.

### Issue 4 — Blank Audio Files (Carried from Epic 3)
**What happened:** The bell MP3 files for the SoundManager were AI-generated as structurally valid but silent files. The issue went undetected because no test checks file content — only file presence. Lord Farquaad discovered the issue manually and has replaced the files with valid bell tolls.

**Root cause:** Binary asset quality is outside the scope of text/DOM-based test tooling. The `validate-data.ts` prebuild hook validated JSON data but not audio files.

**Prevention:** Add a minimum file-size check to `validate-data.ts` for audio assets (any `src/assets/audio/*.mp3` under ~1KB is likely blank/invalid). This provides a fast-fail at build time rather than a silent runtime failure.
