## Starter Template Evaluation

### Primary Technology Domain

Vue 3 SPA/PWA — single-page application with PWA capabilities, no backend, static deployable, Chrome-primary. Component model matches the UX spec's Layer 1–4 architecture directly.

### Starter Options Considered

| Option | Verdict |
| --- | --- |
| `create-vue` (official) + manual additions | ✅ Selected — official, maintained, exact control |
| `vite-vue-ultimate-starter` | Too opinionated (includes daisyUI, unnecessary for bespoke design) |
| `@vite-pwa/create-pwa` | PWA-focused but requires more manual Vue setup |

### Selected Starter: `create-vue` (official Vue scaffolding tool)

**Rationale:** Official Vue project generator. Gives us TypeScript, Vitest, and Playwright via CLI flags with proper configuration. Tailwind CSS v4 and vite-plugin-pwa are added immediately after — both have straightforward Vite plugin integration. Full control, zero imposed aesthetic.

**Initialization Sequence:**

```bash

npm install -D tailwindcss @tailwindcss/vite

```

**Architectural Decisions Provided by Starter:**

**Language & Runtime:**
TypeScript strict mode. Vue 3 (v3.5.x stable). Vite 8 with Rolldown bundler (10-30x faster builds).

**Styling Solution:**
Tailwind CSS v4 via `@tailwindcss/vite` plugin. Zero PostCSS config required. Design tokens (colors, typography, spacing, animation durations) defined via `@theme` in a single CSS file — replaces `tailwind.config.js` as the token source of truth.

**Build Tooling:**
Vite 8 + Rolldown. Static output — all assets are build-time artifacts. PWA manifest and service worker generated by `vite-plugin-pwa` (Workbox-backed).

**Testing Framework:**

- Vitest — unit tests (game engine, date logic, elimination algorithm) and component tests
- Playwright — E2E tests for full ritual sequences and offline PWA behavior
- Split is clean: Vitest owns correctness, Playwright owns ritual flow

**State Management:**
Pinia — game state store (board state, current guess, hard mode, settings), persistence store (streak, history, localStorage adapter).

**Code Organization:**

```text
src/
  components/     # UI components (Tile, Board, FunnelChart, EtymologyCard, etc.)
  composables/    # Game engine logic, elimination algorithm, sound management
  stores/         # Pinia stores (game, persistence, settings)
  data/           # Static word list and etymology JSON (build-time artifacts)
  assets/         # Audio clips, fonts fallback
```

**Development Experience:**
HMR via Vite. Vue DevTools. ESLint + Prettier. TypeScript strict mode catches data integrity issues at compile time.

**Note:** Project initialization is the first implementation story. The elimination algorithm composable is the second — it must be unit-tested before any UI is built.

## Project Context Analysis

### Requirements Overview

**Functional Requirements:**

27 FRs across 6 domains: Daily Puzzle (FR1–FR8), Post-Solve Experience (FR9–FR13), Word & Etymology Data (FR14–FR17), Streak & Progress (FR18–FR21), Offline & Persistence (FR22–FR25), Settings (FR26–FR27), with Phase 2 analytics (FR28–FR31) explicitly post-MVP.

Architecturally, the FRs establish three distinct data pipelines: (1) the Wordle elimination algorithm that must reduce the valid word space after each guess (core of the funnel, most complex piece); (2) the static data pipeline for word list and etymology JSON compiled at build time; (3) local persistence for all player state with corruption-safe read/write.

**Non-Functional Requirements:**

- Performance: First load ≤2s, subsequent (offline) ≤500ms, tile flip ≤100ms, post-solve render ≤300ms. Zero network requests during gameplay.
- Accessibility: WCAG AA (4.5:1 contrast) for all tile states and deuteranopia palette. Full keyboard operability.
- Reliability: Player data survives refresh, tab close, and browser restart. Mid-puzzle state fully restored on reload.
- Data integrity: Corrupted/unavailable storage must display explicit error and clean initial state — never silent corruption or incorrect state.
- Timezone: Date-deterministic word selection must produce the same answer regardless of device timezone offset (NFR13).

**Scale & Complexity:**

Single-user, no backend, no auth, no multi-tenancy. All complexity is local: algorithm correctness, data pipeline integrity, PWA caching strategy, and animation orchestration timing.

- Primary domain: Web SPA/PWA, greenfield, Chrome-primary
- Complexity level: Low (infrastructure) / Medium (algorithm + orchestration)
- Estimated architectural components: ~12 UI components, 3 data layers (static build-time data, service worker cache, localStorage), 1 game engine module, 1 post-solve orchestration layer

### Technical Constraints & Dependencies

- **No backend, ever** — no server, no API calls at runtime; any data must exist at build time or in localStorage
- **Static deployable** — GitHub Pages / Netlify with no server-side config; all assets are build artifacts
- **Chrome desktop primary** — other browsers graceful degradation only
- **Google Fonts** — Inter + Lora must be cached by the service worker for offline etymology card rendering
- **Build-time data compilation** — word list (valid guesses + answer sequence) and etymology JSON must be validated at build time; missing etymology entries must surface as build warnings, not runtime surprises
- **Wordle word list** — use the well-documented open-source Wordle word list; the elimination algorithm must be tested independently before any UI is built

### Cross-Cutting Concerns Identified

1. **Offline caching strategy** — service worker must cache app shell, word list JSON, etymology JSON, and Google Fonts; cache invalidation strategy needed for future word list updates
2. **Timezone-safe date logic** — puzzle index derived from date must normalize to UTC or a fixed offset to guarantee same answer across timezones
3. **Wordle elimination algorithm** — reduces valid word space after each guess using letter-position constraints; must be correct before funnel UI is built; drives the core innovation
4. **Local state persistence** — single localStorage schema owns all player state; must handle first-run (no data), mid-puzzle restore, and corruption gracefully
5. **PostSolveTransition orchestration** — sound-leads-visuals sequencing (bowl ring → board dim → funnel fade → etymology fade) has precise timing dependencies; this is architectural, not just CSS
6. **WCAG AA accessibility** — color is never sole feedback signal; aria-labels required on all tile states; deuteranopia palette toggled live without page reload

## Core Architectural Decisions

### Decision Priority Analysis

### Data Architecture

#### localStorage Schema — Keyed Entries

Each concern is an independent key. Corruption in one key cannot affect others. All keys prefixed `myrdle_` for debuggability in DevTools.

| Key | Value | Notes |
| --- | --- | --- |
| `myrdle_settings` | `{ hardMode: boolean, deuteranopia: boolean }` | User preferences |
| `myrdle_streak` | `{ count: number, lastSolvedDate: string }` | Date as `YYYY-MM-DD` UTC |
| `myrdle_game_YYYY-MM-DD` | `{ guesses: string[], solved: boolean, funnelData: number[] }` | One key per calendar day |

On read, each key is independently validated. If any key is missing or corrupt, the app initializes that key to its default state and surfaces an error message per NFR12 (never silent corruption).

#### Static Data Files — Separate JSON at Build Time

Three separate static assets compiled at build time:

| File | Contents | Size est. |
| --- | --- | --- |
| `src/data/valid-words.json` | Array of ~12,000 valid 5-letter guess words | ~80KB |
| `src/data/answers.json` | Ordered array of ~2,300 answer words (index = day offset from epoch) | ~16KB |
| `src/data/etymology.json` | Object keyed by word: `{ pos, definition, origin }` | ~500KB |

Etymology file is the largest asset — service worker caches it on first load. A build-time validation script confirms every word in `answers.json` has an entry in `etymology.json` (missing entries surface as build warnings, not runtime surprises).

### Frontend Architecture

#### Game Engine — Composable + Thin Pinia Store

`useGameEngine` composable owns all game logic as pure TypeScript functions: word validation, tile state derivation, elimination algorithm, hard mode constraint checking, date-to-answer-index mapping (UTC-normalized per NFR13). Zero Vue dependencies — fully unit-testable with Vitest.

Pinia `useGameStore` owns reactive state only: current board state, active row, game phase (`playing` / `won` / `lost`), current day's funnel data. It calls into `useGameEngine` composable functions — it does not contain logic.

This split means the elimination algorithm and date logic can be tested in isolation before a single component is built.

#### PostSolveTransition — Orchestration Composable

`usePostSolveTransition` composable sequences the ritual: sound trigger → board dim → funnel fade-in → etymology fade-in → keyboard navigation. Timing constants (300ms pause, 500ms dim, 400ms fade, 4s auto-advance) live here as named constants, not scattered in component CSS.

### Authentication & Security

Not applicable. No user accounts, no backend, no network requests during gameplay. All data is local.

### API & Communication Patterns

No API. All data is static JSON imported at build time and cached by the service worker. Zero runtime network requests after first load (NFR5).

### Infrastructure & Deployment

**Hosting:** Local only. App runs via `npm run dev` during development. Production build via `npm run build` produces a static `dist/` folder — portable, deployable to static hosting in the future with zero config changes.

**CI/CD:** None. Tests run manually:

```bash

npm run test:e2e     # Playwright — full ritual flow, offline behavior
```

The elimination algorithm and date-determinism tests should be run before any game-visible feature is considered done.

## Implementation Patterns & Consistency Rules

### Naming Patterns

**Code Naming Conventions:**

| Construct | Convention | Example |
| --- | --- | --- |
| Vue components | PascalCase filename + `<script setup>` | `GameTile.vue`, `FunnelChart.vue` |
| Composables | camelCase, `use` prefix | `useGameEngine.ts`, `usePostSolveTransition.ts` |
| Pinia stores | camelCase, `use` + `Store` suffix | `useGameStore.ts`, `useSettingsStore.ts` |
| TypeScript interfaces | PascalCase, no `I` prefix | `GameState`, `TileState`, `EtymologyEntry` |
| TypeScript enums | PascalCase name, SCREAMING_SNAKE values | `GamePhase.PLAYING`, `GamePhase.WON` |
| Constants | SCREAMING_SNAKE_CASE | `MAX_GUESSES`, `WORD_LENGTH`, `AUTO_ADVANCE_MS` |
| CSS design tokens | kebab-case with `--` prefix in `@theme` | `--color-tile-correct`, `--duration-tile-flip` |
| localStorage keys | `myrdle_` prefix, snake_case | `myrdle_streak`, `myrdle_settings` |
| Test files | co-located, `.test.ts` suffix | `useGameEngine.test.ts`, `GameTile.test.ts` |

Data file naming: `valid-words.json`, `answers.json`, `etymology.json` — all kebab-case, in `src/data/`.

### Structure Patterns

**Component internal structure — `<script setup>` ordering:**

1. Imports
2. Props / emits definitions
3. Store refs
4. Local reactive state
5. Computed properties
6. Functions / event handlers
7. Lifecycle hooks (`onMounted`, `onUnmounted`)

**Composable return shape:** Always return a plain object so destructuring is named:

```ts
// ✅ correct
const { isValid, tileStates } = useGameEngine()
// ❌ wrong — array destructuring loses names
const [isValid, tileStates] = useGameEngine()
```

**Pinia store style:** Composition style (`defineStore` with `setup` function) for all stores — consistent with `<script setup>` mental model:

```ts
export const useGameStore = defineStore('game', () => {
  // refs = state, computed = getters, functions = actions
})
```

### Format Patterns

**TypeScript strict mode, no `any`:** `any` is banned. Use `unknown` + type guards at all data boundaries (localStorage reads, JSON parsing).

**Date format:** All dates stored and compared as `YYYY-MM-DD` UTC strings. Puzzle index computed by diffing against a fixed epoch in UTC — never using local `Date` methods directly.

**Tile state type:**

```ts
type TileState = 'empty' | 'filled' | 'correct' | 'present' | 'absent'
```

Never raw strings for tile states — always this union type.

**Funnel data shape:** `number[]` — index = guess number (0-based), value = valid words remaining after that guess. Length equals number of guesses submitted (1–6).

### Communication Patterns

**No direct store mutation from components:** Components call store actions only. State transitions go through named action functions — never direct property assignment from a component.

**Game phase transitions:** Only `useGameStore` actions may change `gamePhase`. No component sets phase directly.

**Sound trigger:** `SoundManager` called exclusively from `usePostSolveTransition` — never from components or game store. Sound is an orchestration concern, not a game state concern.

**localStorage access:** Only `usePersistenceStore` touches `localStorage`. No other composable or component reads/writes storage directly.

### Process Patterns

**localStorage corruption handling:**

```ts
function safeRead<T>(key: string, fallback: T): T {
  try {
    const raw = localStorage.getItem(key)
    if (!raw) return fallback
    return JSON.parse(raw) as T
  } catch {
    return fallback // surface error via storageError flag in usePersistenceStore
  }
}
```

Every localStorage read uses this pattern. Errors surface as a reactive `storageError` flag — components read this to show the NFR12 error message.

**Invalid word feedback:** Row shake only — no sound, no color, no text. Shake CSS class applied then removed after 300ms. No other mechanism.

**Post-solve auto-advance:** The 4-second timer in `usePostSolveTransition` is cancellable. If user presses `Space`/`Enter`, timer cancels and transition fires immediately. Never double-fires.

### Enforcement Guidelines

All AI agents MUST:

- Use `<script setup>` syntax — Options API is banned
- Use Pinia composition-style stores
- Place timing constants in `usePostSolveTransition` — never inline in CSS or templates
- Read/write localStorage only through `usePersistenceStore`
- Use `TileState` union type — never raw strings for tile states
- Run `npm run test:unit` after any game engine change

Anti-patterns to reject:

- `any` type — use `unknown` + type guards
- Direct `localStorage` access outside `usePersistenceStore`
- Game logic inside Vue components — belongs in composables
- Hardcoded timing values in component templates or CSS — use named constants
- Options API — project is `<script setup>` only
