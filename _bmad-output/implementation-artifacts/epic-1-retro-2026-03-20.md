The pure TypeScript game engine composable (zero Vue imports) made Story 1.3 trivially testable and set a pattern that every subsequent story followed cleanly. The localStorage-only-in-persistence-store rule, established in Story 1.4, is already shaping Epic 2 component design for the better.

**Story intelligence handoffs worked.**
Each story carried forward precise facts from the previous one (word counts, file paths, the 'oaken' epoch detail) without requiring re-discovery. This kept implementation clean and avoided rework.

**Data pipeline exceeded minimums with a clean validation hook.**
The prebuild script catches data integrity issues at build time, not runtime. That's exactly the right place.

**41 tests, zero failures.**
Starting with full test coverage on core logic before any UI exists is the correct order of operations.

## Challenges and Struggles

**CLI non-interactive limitation (Story 1.1).**
`npm create vue@latest` cannot run non-interactively in non-TTY environments. The scaffold had to be applied manually from cached template files. Workaround was sound but unexpected friction. Note for any future project initialization: verify CLI tools support non-interactive mode before relying on them.

**npm package quality (Story 1.2).**
The `wordle-wordlist` package had a bug (referenced `cache.guesses` which doesn't exist). Required regex-parsing the source file to extract the actual data. `word-list` was also installed but ultimately unused. Lesson: check package quality before adopting — install count is not a quality signal.

**Dev Notes test table errors (Story 1.3).**
The duplicate letter algorithm test table in the story spec had incorrect expected values for all three test cases (SPEED/SPELL, AABBB/BAAAA, ABBEY/KEBAB). The dev agent had to manually trace the two-pass algorithm to produce correct results. Lesson: when spec documents contain computed expected values, they must be verified against the algorithm, not taken on faith.

**BMAD Dashboard '&' parsing bug.**
Story titles containing '&' (e.g. "Game Engine Composable and Algorithm Tests") caused display issues in the BMAD Dashboard extension. Bug has been reported to the dashboard repo. Workaround adopted immediately: use 'and' in all story and epic display names going forward.

**Inconsistent sprint-status.yaml updates.**
Status transitions (in-progress → review → done) were not consistently applied during implementation. Made it harder to read true epic state from the dashboard. Root cause: status updates felt like administrative overhead rather than part of the definition of done.

**Approval fatigue — `node -e` prompts.**
Repeated VS Code approval prompts for `node -e` commands created friction. Resolved by adding `Bash(node -e:*)` to `.claude/settings.local.json`.

## Key Insights

1. **Verify packages before adopting.** Install count is not a proxy for correctness.
2. **Spec test tables need algorithm verification.** Computed expected values in dev notes can be wrong — trace the algorithm manually.
3. **Status updates are code.** `sprint-status.yaml` is a first-class artifact, not optional cleanup.
4. **Architecture enforcement on story 1 compounds positively.** Every constraint established in Epic 1 (pure TS composable, no direct localStorage, no game logic in components) saves work in every subsequent epic.
5. **Story intelligence handoffs (passing precise facts forward) prevent re-discovery rework.**
