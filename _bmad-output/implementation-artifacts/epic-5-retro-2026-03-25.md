## What Went Well

### CSS-Only Deuteranopia Solution
Story 5.2 used a CSS class toggle on `<html>` with variable overrides — zero component changes required. This worked because Epic 2 established all tile colors as CSS custom properties. Good architecture paying dividends.

### Flexible Story Ordering
Stories 5.5 and 5.6 were prioritized before 5.3 and 5.4 to address etymology data quality early. The stub data ("A five-letter English word.") had been sitting since Epic 1. Reordering worked smoothly because stories were designed with minimal cross-dependencies.

### Test Coverage Growth
60 new unit tests and 15 new e2e tests. E2E test quality improved — assertions check behavioral state (CSS classes, computed values) rather than just DOM presence. This informally applied the "behavior not structure" standard from Epic 4's retro.

### Code Review Effectiveness
Every story that went through review had real issues caught and fixed. The system worked as designed — issues were manageable and fixes were straightforward.

### Clean Delivery
6/6 stories delivered, zero production incidents. Lord Farquaad play-tested successfully (solved SHRUB in 4 guesses). Full quality review conducted with real users (HB scrum team, 5 people).

## What Was Hard

### Accessibility Edge Cases Across Multiple Stories
Stories 5.3 and 5.4 both had accessibility edge cases caught in review:
- Focus restoration on `?`-dismiss needed `onUnmounted` (not inline close handler)
- `stopImmediatePropagation` needed on Escape to prevent post-solve double-dismiss
- Board-dim inline style defeated CSS `!important` reduced-motion override
- aria-live announcer needed `nextTick` clear-then-set for screen reader reliability

These weren't careless mistakes — they're the kind of interaction-level bugs that only surface when testing full user flows.

### Contrast Ratio Assumption Was Wrong
The original `#538d4e` (tile-correct green) was assumed WCAG AA compliant. Actual ratio: 4.35:1 — below the 4.5:1 threshold. Had to adjust to `#5a9654` (4.88:1). This was a planning-phase miss: contrast ratios should have been verified when color tokens were defined in Epic 2.

### Story 5.5 Marked Done Before All ACs Committed
Story 5.5 didn't commit the EtymologyCard rendering for new fields — 5.6 had to pick up that work. The paired story design (schema first, data second) blurred the boundary between them.

### Epic 4 Retro Action Items Not Tracked
5 action items from the Epic 4 retrospective went unaddressed — none were embedded in story task lists. They existed only in the retro document and were forgotten.

## Real-User Quality Review Feedback

Quality review conducted 2026-03-24 and 2026-03-25 with the HB scrum team (Jason Knight, Ryan Overton, Adam Kogut, Nicole Trevino, Christopher Corns, Allison Stiff, Debbie Heimmermann).

### Ryan Overton (Colorblind User)
1. **Deuteranopia palette insufficient** — "It still looks green to me. It needs to be better." The 5.2 toggle doesn't provide enough contrast for actual deuteranopia.
2. **No secondary indicator beyond color** — "Until I have a contrasting, I can see the two. I don't know which is which." When only one color state is on screen, he can't tell yellow from green. Needs a legend, icon, or shape indicator.
3. **Board collapse post-solve** — "What would be kind of cool is if it collapses your board to the single row of the correct letter and then that moves up. So you don't have to scroll down."

### Adam Kogut
4. **Keyboard key colors inverted** — "The lighter means that it's still available. But then the darker is not." Used/unused key distinction is counterintuitive in dark mode.
5. **App doesn't respond to OS theme changes** — When switching OS from dark to light mode, the app didn't respond. Needs `prefers-color-scheme` support or a theme toggle.

### Jason Knight (Self-Identified During Demo)
6. **Music auto-play disruptive** — "Let me go ahead and mute this because annoyingly I made it to start playing music." Keypress-triggered music needs replacement with a visible on/off toggle.
