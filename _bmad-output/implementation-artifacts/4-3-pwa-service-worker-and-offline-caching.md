# Story 4.3: PWA Service Worker and Offline Caching

Status: done

## Story

As a player,
I want the app to load instantly and work completely offline after my first visit,
so that my morning ritual is never blocked by network availability.

## Acceptance Criteria

1. **Given** the player visits the app for the first time on a live connection **When** the page finishes loading **Then** the service worker installs silently in the background — no progress indicator, no "installing" UI (FR23, UX Journey 3) **And** `valid-words.json`, `answers.json`, `etymology.json`, and Google Fonts (Inter + Lora) are all cached

2. **Given** the service worker has been installed on a previous visit **When** the player opens the app with no network connection **Then** the app shell loads and is fully playable — board, keyboard, word validation, tile reveals all function (FR22, NFR11) **And** the etymology card displays correctly using cached data

3. **Given** the app loads from the service worker cache **When** measured from navigation start to interactive **Then** load time is within 500ms (NFR2)

4. **Given** `vite.config.ts` PWA configuration **When** reviewed **Then** the cache manifest explicitly includes the app shell, `valid-words.json`, `answers.json`, `etymology.json`, and Google Fonts URLs **And** the service worker is generated by `vite-plugin-pwa` (Workbox-backed)

5. **Given** `npm run preview` serving the production `dist/` **When** Chrome DevTools Network tab is set to Offline after first load **Then** a full page reload serves the app entirely from cache — zero network requests (NFR5, NFR11)

### Complete `vite.config.ts` VitePWA Block (replace the existing VitePWA call)

```ts
VitePWA({
  registerType: 'autoUpdate',
  manifest: {
    name: 'Myrdl',
    short_name: 'Myrdl',
    description: 'Your daily word ritual.',
    theme_color: '#1a1a22',
    background_color: '#1a1a22',
    display: 'standalone',
    icons: [
      {
        src: '/favicon.svg',
        sizes: 'any',
        type: 'image/svg+xml',
        purpose: 'any maskable',
      },
    ],
  },
  workbox: {
    globPatterns: ['**/*.{js,css,html,ico,png,svg,json}'],
    runtimeCaching: [
      {
        urlPattern: /^https:\/\/fonts\.googleapis\.com\/.*/i,
        handler: 'StaleWhileRevalidate',
        options: {
          cacheName: 'google-fonts-stylesheets',
          expiration: {
            maxEntries: 5,
            maxAgeSeconds: 60 * 60 * 24 * 365, // 1 year
          },
          cacheableResponse: { statuses: [0, 200] },
        },
      },
      {
        urlPattern: /^https:\/\/fonts\.gstatic\.com\/.*/i,
        handler: 'CacheFirst',
        options: {
          cacheName: 'google-fonts-webfonts',
          expiration: {
            maxEntries: 20,
            maxAgeSeconds: 60 * 60 * 24 * 365, // 1 year
          },
          cacheableResponse: { statuses: [0, 200] },
        },
      },
    ],
  },
}),
```

`StaleWhileRevalidate` for `fonts.googleapis.com` — serves immediately from cache while refreshing in background (CSS stylesheets can update URLs). `CacheFirst` for `fonts.gstatic.com` — the actual font binary files are immutable (versioned URLs), so never go to network after first cache.

### JSON Data Files: Bundle vs. Separate Assets

All three JSON files are **statically imported** in source:
- `src/composables/useGameEngine.ts` imports `valid-words.json` and `answers.json`
- `src/components/layout/PostSolveTransition.vue` imports `etymology.json`

Vite bundles static JSON imports into the JS chunk. In the `dist/` output they will appear as:
- `valid-words.json` and `answers.json` — inlined into the game engine JS chunk (small enough, ~16KB combined)
- `etymology.json` (~500KB) — Vite may keep this as a separate chunk or inline it

**Adding `json` to `globPatterns`** catches `etymology.json` if Vite outputs it as a separate file. If it ends up inlined in a `.js` chunk, the `**/*.js` pattern already covers it. Either way, all data is cached. The `json` pattern is defensive coverage — it doesn't hurt if there's nothing to match.

No source file changes needed for imports. Do NOT switch from static imports to `fetch()` calls.

### Google Fonts URL

The CSS at `src/style.css` imports:
```css
@import url('https://fonts.googleapis.com/css2?family=Inter:wght@400;500;700&family=Lora:ital,wght@0,400;0,700;1,400&display=swap');
```
The two runtime cache entries above handle:
1. `fonts.googleapis.com` — the CSS stylesheet that lists the font files
2. `fonts.gstatic.com` — the actual `.woff2` binary font files referenced by the stylesheet

Both must be cached for the etymology card (which uses Lora) to render offline.

### Playwright Test Pattern for Service Worker

Playwright supports detecting service workers via `browserContext.serviceWorkers()` and `browserContext.waitForEvent('serviceworker')`. For reliable SW install detection:

```ts
import { test, expect } from '@playwright/test'

test.describe('PWA / offline', () => {
  test('service worker registers on first visit', async ({ page, context }) => {
    // Wait for SW to register
    const swPromise = context.waitForEvent('serviceworker')
    await page.goto('/')
    await page.waitForLoadState('networkidle')
    const sw = await swPromise
    expect(sw.url()).toContain('sw.js')
  })

  test('app loads offline after SW install', async ({ page, context }) => {
    // First visit — populate cache
    await page.goto('/')
    await page.waitForLoadState('networkidle')
    // Let SW activate — wait for it to take control
    await page.waitForTimeout(1000)

    // Go offline
    await context.setOffline(true)

    // Reload — should serve from cache
    await page.reload()
    await page.waitForLoadState('domcontentloaded')

    // App shell should be present
    await expect(page.locator('#app')).not.toBeEmpty()

    await context.setOffline(false)
  })

  test('word validation works offline', async ({ page, context }) => {
    await page.goto('/')
    await page.waitForLoadState('networkidle')
    await page.waitForTimeout(1000)

    await context.setOffline(true)
    await page.reload()
    await page.waitForLoadState('domcontentloaded')

    // Type a valid word and submit
    for (const key of ['c', 'r', 'a', 'n', 'e']) {
      await page.keyboard.press(key)
    }
    await page.keyboard.press('Enter')

    // At least one tile row should have state (means JS ran, engine worked from cache)
    const tiles = page.locator('.tile-row').first().locator('.tile')
    await expect(tiles).toHaveCount(5)

    await context.setOffline(false)
  })
})
```

**Important: These tests require the production build** — the dev server does not run a service worker. The `playwright.config.ts` is already configured for this: in CI it uses `npm run preview` (port 4173), locally it uses `npm run dev` (port 5173).

**Offline tests only work in CI/preview mode.** For local dev, these tests will be skipped or will fail gracefully because the dev server doesn't serve a SW. Use `test.skip(!process.env.CI, 'SW tests require production build')` to guard them, OR run `npm run build && npm run preview` locally.

### `vite-plugin-pwa` v1.2.0 Notes

`vite-plugin-pwa` ^1.2.0 (the version installed) uses Workbox v7 internally. Key points:
- `registerType: 'autoUpdate'` — new SW versions are applied on next navigation, no user prompt. This is the correct behavior for this app (no toast/banner needed).
- `manifest` object in `VitePWA({})` generates `manifest.webmanifest` in `dist/` automatically.
- `workbox.globPatterns` matches against `dist/` output files — patterns are relative to the dist root.
- `runtimeCaching` entries use Workbox Strategy names as strings: `'CacheFirst'`, `'StaleWhileRevalidate'`, `'NetworkFirst'`.
- The plugin generates `dist/sw.js` which is the registered service worker. No manual `sw.js` needed.
