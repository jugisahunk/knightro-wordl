## What Went Well

**Architecture discipline compounded positively.**
Epic 1's guardrails — pure TypeScript composables, store-only-in-GameView, no game logic in leaf components — made every Epic 2 story land cleanly. No story required architectural debate. The pattern was established; everyone followed it.

**All Epic 1 action items honored.**
The 'and' not '&' rule held across all 5 story titles. sprint-status.yaml was updated in real time (inline comments mid-epic confirm this). Three for three on follow-through — first time that's happened.

**Code review caught real bugs in Story 2.2.**
Eight patches were applied: timer leak (setTimeout cleanup on unmount), animation race condition (animationend vs watch timing), aria-label empty-letter guard, shake reduced-motion fallback, and perspective placement. These were real bugs that would have caused production issues. The right time to find them was in review, not in Epic 3.

**Smooth velocity on Stories 2.3–2.5.**
After 2.2's review, the remaining three stories were issue-free or self-correcting (one `getLabel` uppercasing bug in 2.3 caught during implementation). Complexity was front-loaded appropriately.

**Story intelligence handoffs continued working.**
Each story carried precise context forward — test counts, store API signatures, token names — without re-discovery.

## Challenges

**Story 2.2 code review volume — 8 patches.**
The densest story of the epic: CSS 3D animations, stagger timing, ARIA grid structure, and timer management all in one story. High surface area produced high review yield. Not a failure — a signal that complex stories benefit from review.

**Epic done transition is not automatic in the BMAD Dashboard.**
When the last story is marked done, the epic status in sprint-status.yaml does not flip automatically. Requires a manual update. Memory saved; process updated.

**Etymology debt still outstanding.**
1,678 stubs carried from Epic 1. Now has a clear resolution plan (see Action Items).

## Key Insights

1. **Architecture enforcement compounds.** Every constraint from Epic 1 saved work in every Epic 2 story. The investment in discipline on story 1 pays dividends indefinitely.
2. **Dense stories need code review.** 2.2 had the highest complexity and the most review findings. For Epic 3, Story 3.4 (PostSolveTransition) is the equivalent candidate — timers, sequencing, multiple composable interactions.
3. **100% action item follow-through is achievable.** When commitments are specific and small, they get done.
4. **Etymology content quality matters for the ritual.** The EtymologyCard is the emotional close of every puzzle. Real etymonline-format entries (pos, definition, etymological origin chain) are worth sourcing properly.

**A2 — Etymology sourcing script**
- Owner: Dev agent (preparation task before Story 3.3)
- Deadline: Before Story 3.3 (EtymologyCard component) begins
- Success criteria: `etymology.json` contains real etymonline-format entries for all 2,315 answer words; words with no etymonline entry remain empty/null (triggering existing fallback display "No etymology on record for this word.")

- Note: Etymonline is copyrighted — acceptable for private/personal use; if project goes public, licensing or paraphrasing will be required
